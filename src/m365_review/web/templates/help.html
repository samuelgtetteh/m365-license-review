What this tool does

It signs in to a single client Microsoft 365 tenant, reads how many licenses are owned versus actually used, and produces a prioritized license optimization report with estimated monthly and annual savings. It is strictly read-only — it never changes anything.

Running a review, step by step

  1. Choose a profile or add one. Pick a saved client from the dropdown, or add a new one by giving it a name plus the Azure client ID — it’s saved so next time you just pick the name.
  2. Choose options on the home page: output formats and whether to include experimental rules.
  3. Sign in. Click “Sign in to a client tenant”. You’ll be sent to Microsoft — sign in as a client tenant admin. First time for a tenant, you’ll approve a read-only consent screen.
  4. Confirm the tenant. Check the tenant name and ID shown match the client you intend to audit, then click “Proceed”.
  5. Wait for the audit. A progress bar shows each step (reading identity, licenses, users, analyzing, writing).
  6. Download. On the results page, download any format or a zip of all of them.

The tenant-confirmation step is deliberate — it prevents accidentally auditing the wrong client.

Output formats

  • Excel (.xlsx) — summary, a License Optimization sheet, license inventory, per-rule detail sheets, and a hidden raw-data sheet.
  • Word (.docx) — a client-facing narrative report led by the optimization summary.
  • JSON — the full structured result for automation or record-keeping.

Files are named {tenant}_{date}.{ext}.

What we check (v1 rules)

RuleFinding
R1Licenses assigned to disabled user accounts.
R2Licenses on users with no sign-in in 90 days (new accounts under 30 days are excluded). Needs Azure AD P1; without it, affected users are listed for manual review instead.
R3Unassigned purchased licenses — seats paid for but not assigned.

Savings are estimates from the price list in config/sku_prices.yaml. Always confirm before acting.

Using the command line

The CLI is an optional alternative to this web page — handy for scripting or automation. It runs inside the Docker container, not on your computer directly, so m365-review is not a command on your host shell (running it there gives “term not recognized”). Invoke it through docker compose run instead.

Step by step

  1. Open a terminal in the project folder (where docker-compose.yml lives).
  2. Prefix each command with docker compose run --rm m365-review, e.g.: 
    docker compose run --rm m365-review m365-review run --profile "Contoso"
  3. Because the container has no browser, sign-in switches to device code automatically: it prints a short code and the URL https://microsoft.com/devicelogin. Open that in any browser, enter the code, and sign in as the client tenant admin.
  4. Confirm the tenant when prompted; the audit runs. Reports are written to the reports/ folder and saved profiles live in data/ (both persist on your machine).

Common commands

# run an audit (all formats) reusing a saved profile
docker compose run --rm m365-review m365-review run --profile "Contoso"

# choose formats / include experimental rules
docker compose run --rm m365-review m365-review run --format xlsx --format json
docker compose run --rm m365-review m365-review run --enable-experimental-rules -v

# list tenant SKUs and flag any missing a price
docker compose run --rm m365-review m365-review sku-check --profile "Contoso"

# manage saved client profiles
docker compose run --rm m365-review m365-review profiles list
docker compose run --rm m365-review m365-review profiles add "Contoso" -c <client-id>

Prerequisites

  • Device-code sign-in requires “Allow public client flows: Yes” on the app registration (Authentication → Settings tab). If you see AADSTS700016 or AADSTS7000218, enable that toggle.
  • Device code also needs the client tenant named up front (pass --tenant contoso.com or a tenant ID, or save it on the profile). Without it you'll get AADSTS50059.
Some tenants block device-code sign-in. If, after a successful sign-in, you land on a Microsoft page saying “You don't have access to this… an authentication flow that is restricted by your admin,” the client's Conditional Access policy blocks the device-code flow. That's a tenant security setting, not a tool problem. Use the web app (this page's normal browser sign-in isn't device code, so it isn't blocked), or install the CLI locally (below) to sign in with a browser.

Running it on your own machine (optional, and the fix for blocked device code)

To have m365-review as a real command on your computer, install the package into Python 3.11+:

pip install -e .
m365-review run --profile "Contoso"

Installed locally it uses your real browser to sign in, so no device code is needed.

Privacy, security & the read-only guarantee

  • Only these read-only Graph scopes are requested, and only GET calls are ever made: {% for s in scopes %}{{ s }}{% if not loop.last %}, {% endif %}{% endfor %}.
  • Your access token is held in memory for the run only — never logged, saved, or put in a report — and is discarded when the audit finishes.
  • Reports can contain user names, UPNs, and sign-in dates. Treat them as sensitive and store them somewhere access-controlled.
  • If usage-report names look scrambled, the tenant has “concealed names” on; ask the client admin to turn it off for per-user detail.

Troubleshooting

“No reply address” (AADSTS500113) on sign-in
Add {{ redirect_uri }} to the app registration under Mobile and desktop applications (not Web), and set “Allow public client flows” to Yes.
“client_secret required” (AADSTS7000218)
The redirect URI was registered under the Web platform. Delete it there and re-add it under Mobile and desktop applications — this app is a public client with no secret.
“No tenant-identifying information” (AADSTS50059) on the CLI
Device-code sign-in needs the tenant named up front. Re-run with --tenant <domain-or-id>, or save it on the profile.
“You don't have access to this” after a successful CLI sign-in
The client tenant's Conditional Access policy blocks the device-code flow. Use the web app (its browser sign-in isn't device code), or install the CLI locally to sign in with a browser. This is a tenant security setting, not a tool issue.
403 during the audit
A required scope wasn’t consented. Have the client admin sign in again and approve the consent screen.
“Client ID not configured”
Set AZURE_APP_CLIENT_ID in .env, or paste it on the home page before signing in.
Sign-in activity missing
The tenant lacks Azure AD P1. R2 degrades to a manual-review list.
